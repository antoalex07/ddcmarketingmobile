**Temp ID management:** Use module-level `nextTempId` starting at -1, decrementing for each new local appointment. On app restart, query `SELECT MIN(appoint_id) FROM appointments WHERE appoint_id < 0` and set `nextTempId` to one less than the minimum (or -1 if no negative IDs exist). This prevents ID collision across app restarts.

**Exported functions:**

**Part A: Create `src/services/AppointmentUploader.js`** mirroring `src/services/LocationUploader.js`.

Constants:
- `MAX_RETRIES = 3`
- `RETRY_DELAY_MS = 1000`
- `delay` helper (same as LocationUploader)

Export `uploadUnsyncedAppointments(token)`:
1. Call `getUnsyncedAppointments()` from appointmentDB
2. If empty, return `{ uploaded: 0, failed: 0 }`
3. Process ONE AT A TIME (per user decision â€” not batched)
4. For each appointment, retry up to MAX_RETRIES:
   a. Transform local record to backend payload format â€” map all `appoint_*` fields from the SQLite row to the POST body. Include: appoint_userid, appoint_staffname, appoint_clientautoid, appoint_clientname, appoint_clientaddress, appoint_clienttype, appoint_appointmentdate, appoint_timefrom, appoint_timeto, appoint_notes, appoint_latitude, appoint_longitude, appoint_status, appoint_report, appoint_reportdate, appoint_followupdate, appoint_followuptimefrom, appoint_followuptimeto, appoint_crtdby, appoint_crton
   b. Call `appointmentService.createAppointment(token, payload)`
   c. On success: call `updateAppointmentId(appointment.id, response.data.appoint_id)` then `markAsSynced(appointment.id)`, increment `uploaded`
   d. On failure: wait `RETRY_DELAY_MS * attempt` ms (linear backoff matching location pattern), retry
5. After all retries exhausted for an appointment: call `incrementRetryCount(appointment.id)`, increment `failed`
6. Return `{ uploaded, failed }`

**Part B: Extend `src/services/AppointmentService.js`** â€” Add `createAppointment` method to the existing `appointmentService` object:

```javascript
createAppointment: async (token, appointmentData) => {
  try {
    const response = await api.post('/appointment/createappointment', appointmentData, {
      headers: { Authorization: `Bearer ${token}` },
    });
    return { success: true, data: response.data };
  } catch (error) {
    return {
      success: false,
      message: error.response?.data?.message || error.message || 'Failed to create appointment',
    };
  }
}
```

Note: The endpoint path `/appointment/createappointment` follows the existing pattern of `/appointment/getallappointments` and `/appointment/updateappointmentstatus` already in AppointmentService.js. If the backend uses a different path, the executor should check but this is the most likely convention.

The uploader should call `api.post` directly (like LocationUploader does) rather than going through AppointmentService for the upload loop â€” this keeps the retry/error handling clean. AppointmentService.createAppointment is for direct online creation from other parts of the app.
